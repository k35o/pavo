# Walkthrough 観点

レビューの指摘とは別に、summary（Review の `body`）の冒頭へ変更全体の walkthrough を追加出力するための指示。
チェック項目を増やすものではなく、他の観点による指摘の内容・rubric には影響しない。

## 出力

- **出力位置と折りたたみ**
  - Review の `body` の先頭に walkthrough を置き、その後に通常の要約・所感を続ける
  - walkthrough 全体を `<details><summary>Walkthrough</summary>` 〜 `</details>` で折りたたむ
  - `<details>` の開始タグの後と `</details>` の前に空行を入れる（空行がないと内部のテーブルや図がレンダリングされない）
- **省略の判断**
  - 変更ファイルが 3 個以下の小さな PR では walkthrough 自体を省略してよい
  - 省略した場合も、通常の summary（確認した観点と結論）は従来どおり書く
- **1. 目的と全体像（2〜3 行）**
  - この PR が何を・なぜ変えるのかを、diff と PR タイトル・説明文から読み取って 2〜3 行で書く
  - 実装手段の列挙ではなく「変更後に何ができるようになるか / 何が直るか」を主語にする
  - PR 説明文と diff の内容が食い違う場合は、diff から読み取れる事実を優先する
  - 「なぜ」が PR タイトル・説明文から読み取れない場合は「何を」の要約にとどめる
- **2. ファイル別の変更概要テーブル**
  - `| ファイル | 変更内容 |` の 2 列テーブルで変更ファイルを網羅する
  - 密接に関連するファイル（実装とそのテスト、コンポーネントとそのストーリー等）は 1 行にグループ化してよい
  - 変更ファイルが多く 1 ファイル 1 行では長くなりすぎる場合は、ディレクトリや機能単位でまとめて 1 行にしてよい
  - 変更内容は「何をしたか」を 1 文で書く。diff の再掲や増減行数の報告はしない
  - lockfile・自動生成ファイルは個別に説明せず「自動生成の更新」等で 1 行にまとめる
- **3. 相互作用図（非自明な場合のみ）**
  - モジュール間の呼び出し関係や処理の流れが diff から自明でない場合のみ、`mermaid` 言語指定のフェンス付きコードブロックで `sequenceDiagram` または `flowchart` を出す（GitHub がネイティブレンダリングする）
  - 描く目安: 3 つ以上のモジュールをまたぐフローの追加・変更、非同期・イベント駆動の連携、外部サービスとのリクエスト往復
  - 省く目安: 単一モジュール内のリファクタ、設定・依存の更新、既存フローの微修正。図がテーブルの言い換えにしかならないなら描かない
  - 登場要素は変更に関係するものだけに絞り、10 ノード程度を上限の目安にする
- **書き方**
  - walkthrough は事実の要約に徹し、指摘・評価・改善提案を混ぜない（それらは inline コメントで行う）
  - diff と PR 情報から確認できたことだけを書き、目的や背景を推測で補わない
